# CarePlan Generation System — Design Doc

**Version:** 0.1  
**Status:** Draft  
**Last Updated:** 2026-05-01

---

## 1. Background & Problem Statement

A specialty pharmacy (CVS) requires pharmacists to manually create care plans for patients on specialty medications. This process takes **20–40 minutes per patient**, is required for Medicare compliance and pharma reimbursement, and the team is severely understaffed with a growing backlog.

**Goal:** Build a web-based tool that allows CVS medical staff to input patient information and automatically generate a professional care plan using an LLM.

---

## 2. Users

- **Primary user:** CVS medical assistants / pharmacists
- **Not a user:** Patients (they receive a printed copy of the generated care plan)

---

## 3. Functional Requirements

### 3.1 Web Form (Input)

The system must accept the following inputs via a web form:

| Field | Type | Validation |
|---|---|---|
| Patient First Name | string | Required, letters only |
| Patient Last Name | string | Required, letters only |
| Patient MRN | string | Required, unique 6-digit number |
| Patient Date of Birth | date | Required |
| Patient Primary Diagnosis | ICD-10 code | Required, valid ICD-10 format |
| Additional Diagnoses | list of ICD-10 codes | Optional, valid ICD-10 format |
| Referring Provider Name | string | Required |
| Referring Provider NPI | string | Required, exactly 10 digits |
| Medication Name | string | Required |
| Medication History | list of strings | Optional |
| Patient Records | string or PDF | Required |

### 3.2 Care Plan Generation

- **One care plan per order (one medication)**
- Generated by LLM based on patient inputs
- Output must include:
  1. Problem list / Drug Therapy Problems (DTPs)
  2. Goals (SMART)
  3. Pharmacist interventions / plan
  4. Monitoring plan & lab schedule
- User can **download** the care plan as a text file

### 3.3 Duplicate Detection Rules

| Scenario | Behavior | Reason |
|---|---|---|
| Same patient + same medication + **same day** | ❌ ERROR — block submission | Definitely a duplicate submission |
| Same patient + same medication + **different day** | ⚠️ WARNING — user can confirm and continue | May be a refill |
| MRN same + name or DOB different | ⚠️ WARNING — user can confirm and continue | Possible data entry error |
| Name + DOB same + MRN different | ⚠️ WARNING — user can confirm and continue | Possibly the same person |
| NPI same + Provider name different | ❌ ERROR — must be corrected | NPI is a national unique identifier |
| NPI same + Provider name same | ✅ Reuse existing provider record | Same provider, no action needed |
| MRN same + name and DOB same | ✅ Reuse existing patient record | Same patient, no action needed |

### 3.4 Reporting & Export

- Quick export of order data for pharma reporting

## 4. Non-Functional Requirements (Production-Ready)

| Requirement | Description |
|---|---|
| Input validation | Every field validated before processing |
| Data integrity | Database-level constraints enforce consistency |
| Error safety | Errors are safe, clear, and contained — no stack traces or PHI exposed to users |
| Modularity | Code is organized and navigable (layered architecture) |
| Test coverage | Critical business logic covered by automated tests |
| Runnable out of the box | Clone repo → one command → system is running (Docker) |

## 5. System Overview (High Level)

```
CVS Medical Staff
      │
      │  fills out web form
      ▼
 ┌─────────────┐
 │   Frontend  │  React
 │  (Web Form) │
 └──────┬──────┘
        │ POST /api/orders
        ▼
 ┌─────────────┐
 │   Backend   │  Python / Django / Django REST Framework
 │   (API)     │  - Input validation
 │             │  - Duplicate detection
 │             │  - LLM call (Claude / OpenAI)
 └──────┬──────┘
        │
        ▼
 ┌─────────────┐
 │  Database   │  PostgreSQL
 │             │  - Patient, Provider, Order, CarePlan tables
 └─────────────┘
```

## 6. Data Model (Preliminary)

```
Patient
  - id (PK)
  - first_name
  - last_name
  - mrn (unique)
  - date_of_birth

Provider
  - id (PK)
  - name
  - npi (unique)

Order
  - id (PK)
  - patient_id (FK → Patient)
  - provider_id (FK → Provider)
  - medication_name
  - primary_diagnosis (ICD-10)
  - additional_diagnoses (list)
  - medication_history (list)
  - patient_records (text)
  - created_at

CarePlan
  - id (PK)
  - order_id (FK → Order)
  - content (text)
  - status (pending / processing / completed / failed)
  - created_at
```

## 7. API Design (Preliminary)

| Method | Endpoint | Description |
|---|---|---|
| POST | `/api/orders/` | Submit patient info, trigger care plan generation |
| GET | `/api/orders/{id}/` | Get order status and care plan content |
| GET | `/api/orders/{id}/download/` | Download care plan as text file |
| GET | `/api/orders/?export=csv` | Export orders for pharma reporting |
